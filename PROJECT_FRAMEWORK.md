# Project Framework And Guidance

Use this document as the operating standard for future projects in this folder.

## 1. Purpose

Define what the project is trying to achieve, who it serves, and what success looks like.

## 2. Project Lifecycle

### Discovery

- Clarify the problem to solve.
- Identify stakeholders and owners.
- List known constraints, dependencies, and assumptions.
- Capture open questions before committing to delivery.

### Planning

- Define scope in and out.
- Break work into milestones or phases.
- Identify risks, blockers, and mitigation options.
- Confirm timeline, responsibilities, and approval points.

### Delivery

- Track progress against the plan.
- Document major decisions and scope changes.
- Surface issues early.
- Keep implementation notes simple and current.

### Handoff

- Summarize what was delivered.
- Note remaining gaps or deferred work.
- Record operating instructions, owners, and support details.
- Capture lessons learned for reuse in the next project.

## 3. Minimum Required Documents

Each project should have at least:

- A brief
- A plan
- Meeting or discussion notes as needed
- A handoff or closeout document

## 4. Standard Sections For Every Project

Each project document should make space for:

- Project name
- Owner
- Contributors
- Date created
- Last updated
- Status
- Objective
- Scope
- Risks
- Dependencies
- Decisions
- Next steps

## 5. Status Definitions

- `Not Started`: work has not begun
- `In Progress`: work is underway
- `Blocked`: progress is paused by a dependency or issue
- `On Hold`: deprioritized or paused intentionally
- `Completed`: delivered and closed

## 6. Decision Logging Guidance

When documenting a decision, always include:

- Date
- Decision made
- Reason
- Owner
- Impact

## 7. Risk Logging Guidance

When documenting a risk, always include:

- Description
- Likelihood
- Impact
- Owner
- Mitigation
- Current status

## 8. Change Control Guidance

If scope changes during a project, record:

- What changed
- Why it changed
- Who approved it
- Timeline or cost impact
- Any new dependencies or risks

## 9. Definition Of Done

A project is ready to close when:

- Core scope has been delivered or formally deferred
- Key decisions are recorded
- Open risks are visible
- Handoff notes are completed
- Owners for next-step support are clear

## 10. Working Principles

- Keep documentation practical and usable.
- Prefer clarity over completeness.
- Write for the next person who joins the project.
- Update documents as work happens, not only after the fact.
